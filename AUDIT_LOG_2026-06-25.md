- Frontend `SOSButton.jsx` no longer sends the `message` key at all when empty.

## 3. Mobile navbar — overflow blocking login (Task #20)

**Reported problem (with screenshot):** on phone, the navbar overflowed off-screen, blocking access to Sign In and other content entirely.

**Root cause confirmed independently:** `Navbar.jsx` had no responsive breakpoint or hamburger menu. The one apparent mobile mechanism, a `hidden-mobile` CSS class, was dead code — confirmed via grep across every `*.css` file in the repo returning zero matches.

**Fix, round 1:** `Navbar.jsx` rewritten (257 → 404 lines) with `isMobile` state (`window.innerWidth < 768` + resize listener), hamburger toggle, full mobile dropdown drawer.

**Your retest screenshots showed this was incomplete:**
- `/landing` was still fully broken — no hamburger at all.
- `/portal` hamburger worked, but the open drawer was visually sliced by the map's search box and IMD legend.

**Root cause of `/landing` failure, found independently via grep:** `LandingPage.jsx` had its own separate hardcoded `<nav>` block, entirely independent of `Navbar.jsx` — the fix to the shared component never touched it. When told to fix this, the engineer's own follow-up grep (which I required) found **two more duplicate navs** in `LoginPage.jsx` and `SignupPage.jsx`. All three replaced with the shared `<Navbar />` component.

**Root cause of `/portal` drawer slicing, found independently via grep:** `PublicPortal.jsx` had floating elements at `z-[1000]` (IMD legend) and `zIndex: 900`, both higher than the navbar's old implicit z-index of 50. Engineer instructed to grep **every** z-index value in the codebase first (found `z-[9999]` PWA banner, `z-[2000]` PinSearch as the highest), then set both the navbar `<header>` and its mobile dropdown to `zIndex: 10000`.

**Final verification (independent, via direct file reads + repo-wide grep):**
- `Navbar.jsx` header and dropdown both confirmed `zIndex: 10000`.
- `LandingPage.jsx`, `LoginPage.jsx`, `SignupPage.jsx` all confirmed to import and render the shared `<Navbar />`, no leftover hardcoded `<nav>`.
- Repo-wide grep for `<nav |<nav>|<header` across all `*.jsx` in `frontend/src` returned **exactly one match: `Navbar.jsx` itself.**

**Status: code-side fix is solid and verified. Awaiting final live retest from you to close Task #20.**

## 5. Mobile bugs found today (register page, map, alert panel, back-link)

### 5a. Registration "Failed to fetch" on phone
**Screenshot evidence:** `/register` on phone showed a red "Failed to fetch" banner.

**Root cause, found independently:** two things, both confirmed by direct file read:
1. `frontend/.env.local` originally had `VITE_SUPABASE_URL=http://127.0.0.1:54321` — loopback, unreachable from a phone on the same network. Engineer updated this to `http://10.117.176.86:54321` (confirmed via direct read of `.env.local`, now correct).
2. `VITE_BACKEND_URL` and `VITE_ML_SERVICE_URL` are empty, which is correct — `backendApi.js` already has a smart fallback to `window.location.hostname`, confirmed via direct read of that file.

### 5b. Map markers/heatmap and "Intelligence Feed" stuck on "Loading live events…"
**You reported this persisted even after the `.env.local` fix above.** I traced the actual data path independently:
- `frontend/src/hooks/useDisasterEvents.js` fetches events via `eventsApi.getAll()` → `backendApi` (REST, port 4000) — this is the correct fallback-aware client, not the broken one.
- **Real root cause, found independently, never touched by the engineer:** `backend/src/app.js` line 33:
  ```js
  const ALLOWED_ORIGINS = (process.env.ALLOWED_ORIGINS || 'http://localhost:3000,http://localhost:5173')
    .split(',').map(o => o.trim()).filter(Boolean);
  ```
  There is no `backend/.env` file setting `ALLOWED_ORIGINS` (confirmed — only `.env.example` has it, with the same restrictive default). The backend's CORS allow-list only accepts `localhost:3000`/`localhost:5173`. Your phone's origin is `http://10.117.176.86:3000`, which is **not in the allow-list**, so the backend rejects every REST request from the phone with a CORS error — this is what shows up as "Failed to fetch" and explains why events/map data never load on mobile, regardless of any frontend fix.
- **Status: identified, fix assigned to engineer (add `ALLOWED_ORIGINS` to `backend/.env` including the LAN origin + restart backend), not yet applied or retested.** This is the most likely true root cause of issue 5b and possibly residual parts of 5a.

### 5c. Intelligence Feed panel covering most of the mobile screen
**Root cause confirmed:** `PublicPortal.jsx` had a hardcoded `width: 300` for the side panel regardless of screen size — on a ~360-400px phone screen that's 75-85% of the viewport.

**Fix verified via direct read:** lines 116-117 now read `isMobile ? 'min(85vw, 320px)' : 300`, gated by a real `isMobile` state + resize listener (lines 66-72). Code is correct.

**You reported this still persists on phone.** Most likely cause: mobile browser serving a cached bundle rather than a logic bug, since the code itself reads correctly. **Engineer instructed to have you hard-refresh / clear cache before concluding otherwise — not yet retested.**

### 5d. No visible way back to citizen portal from `/portal`
**Root cause:** the link existed in code (`Navbar.jsx`, citizen role → `/citizen`) but rendered as plain unstyled text, blending into the surrounding section labels — not visually a button, hence "invisible" in practice.

**Fix verified via direct read:** `Navbar.jsx` lines 340-355 — the citizen NavLink now renders with a `Globe2` icon, green background/border, bold text, and explicit label `← Back to My Portal`. Confirmed `ROLE_NAV` already supplied `icon: Globe2`, so the icon will actually render.

**Status: this fix was not reported as broken in your last retest — treated as resolved pending final confirmation. Task #19 — in progress, awaiting close-out confirmation.**
